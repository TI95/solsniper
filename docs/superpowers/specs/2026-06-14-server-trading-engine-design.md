# Под-проект 1 — Серверный торговый движок + модель данных + шифрование ключей

**Дата:** 2026-06-14
**Статус:** утверждён к реализации
**Фаза:** Фундамент (1 из 5)

## Контекст

`solsniper` — Solana-снайпер. Сейчас торговый движок целиком работает в браузере
(`frontend/src/hooks/useAutoTrade.ts`): покупка/продажа в `useEffect` + `setInterval`,
состояние в `localStorage` и глобальном объекте `purchasedTokens`. Приватный ключ зашит во
фронтенд через `VITE_PRIVATE_KEY` — попадает в собранный JS-бандл (любой может извлечь) и
является единственным глобальным кошельком. В БД сохраняются только покупки
(`TokensData`), без продаж и без PnL.

Цель доработки (вся, разбита на 5 под-проектов): кастодиальная мультипользовательская
платформа, где пользователь привязывает свой кошелёк в профиле, серверный бот торгует 24/7,
всё сохраняется в БД, пользователь видит график profit/loss (реализованный PnL по сделкам +
кривая общей стоимости кошелька), фильтр кандидатов редактируется из UI.

Решения по архитектуре (из брейншторма):
- **Кастодиальная модель:** приватный ключ шифруется и хранится на сервере, бот торгует на
  бэкенде 24/7 даже при закрытом браузере.
- **График PnL:** и реализованный PnL по сделкам, и кривая общей стоимости кошелька (требует
  фонового воркера снапшотов — Под-проект 3).
- **Подход к запуску движка:** B — отдельный воркер-процесс, общий репозиторий и Mongo,
  торговля изолирована от API-процесса.
- **Управление ботом:** кнопка Start/Stop в UI. Ручная продажа сохраняется (обе биржи).

## Разбивка на под-проекты (общий роадмап)

1. **Серверный движок + модель данных + шифрование** ← этот спек.
2. Модель данных — добивается под аналитику (кошельки/позиции/сделки уже здесь; снапшоты — в 3).
3. Аналитика/PnL — воркер снапшотов баланса + эндпоинты + график на фронте.
4. Фильтр (UI + БД) — страница настроек, сохранение per-user, движок читает фильтр из БД.
5. UI кошелька в профиле — расширенное управление кошельком.

Каждый под-проект проходит отдельный цикл спек → план → реализация.

## Объём Под-проекта 1

Перенос торгового движка из браузера на сервер, кастодиальное хранение кошелька на
пользователя с шифрованием, новая модель данных (кошельки/позиции/сделки), серверные
эндпоинты управления, минимальная переделка фронта под новую модель. Точечные фиксы auth,
так как мы и так трогаем этот слой (привязка кошелька).

**Не входит в этот под-проект:** снапшоты баланса и график (3), редактируемый фильтр в UI
(4), расширенный UI управления кошельком (5). Фильтр на этой фазе — константы в серверном
конфиге.

## Компоненты

### 1. Хранение кошелька и шифрование

- Пользователь вставляет приватный ключ в профиле → `POST /wallet`.
- Ключ шифруется **AES-256-GCM**, мастер-ключ берётся из env `WALLET_ENCRYPTION_KEY`
  (32 байта; живёт вне БД — секрет окружения/секрет-менеджер).
- В коллекцию `Wallet` пишутся: `publicKey`, `encryptedSecret`, `iv`, `authTag`.
- Плейнтекст приватного ключа **никогда** не хранится и **никогда** не возвращается клиенту.
  Наружу отдаётся только `publicKey`.
- Расшифровка происходит только в памяти воркера непосредственно перед подписью транзакции.
- Шифрование **серверное** (мастер-ключ), без пароля-на-сессию пользователя — это требование
  24/7-торговли без присутствия пользователя.

**Формат приватного ключа на входе:** принимаем массив байт через запятую (как текущий
`VITE_PRIVATE_KEY`) и base58 (обычный экспорт из Phantom/Solflare). На бэке нормализуем в
`Uint8Array`, валидируем через `Keypair.fromSecretKey`, из неё же берём `publicKey`.

### 2. Модель данных (Mongo / Mongoose)

- **`Wallet`** — `user: ObjectId(User)`, `publicKey: string`, `encryptedSecret: string`,
  `iv: string`, `authTag: string`, `botEnabled: boolean` (default false), `createdAt`.
  Один кошелёк на пользователя в этой фазе (уникальный индекс по `user`).
- **`Position`** (развитие текущего `TokensData`) — открытые позиции: `user`, `tokenAddress`,
  `dexId: 'raydium' | 'pumpswap'`, `amount` (нормализованные токены), `amountRaw`,
  `decimals`, `buyPriceUSD`, `buyPriceSOL`, `totalCostUSD`, `txId`, `status: 'open' |
  'closed'`, `openedAt`. Индекс по `user + status`.
- **`Trade`** — закрытые сделки: `user`, `tokenAddress`, `dexId`, `buyPriceUSD`,
  `sellPriceUSD`, `amount`, `realizedPnlUSD`, `reason: 'take_profit' | 'stop_loss' |
  'manual' | 'dust'`, `txId`, `closedAt`. Индекс по `user + closedAt`. Кормит «график
  сделок» в Под-проекте 3.

Снапшоты баланса и настройки фильтра — отдельные коллекции в Под-проектах 3 и 4.

### 3. Торговый движок (отдельный воркер-процесс)

- Entry: `backend/src/worker/index.ts`, npm-скрипт `npm run worker` (dev — nodemon).
- Главный цикл по активным пользователям (`botEnabled === true` и есть `Wallet`):
  1. Сбор кандидатов — DexScreener boosted + добор по парам (перенос логики из
     `boosted-tokens-api.ts` на бэк).
  2. Применение фильтра (значения — константы серверного конфига на этой фазе):
     мин. ликвидность USD, макс. капитализация, мин. бусты, макс. возраст, разрешённые
     биржи (raydium/pumpswap).
  3. Покупка: соблюдение лимита позиций и кулдауна, маршрут raydium vs pumpswap по `dexId`,
     запись `Position`.
  4. Продажа: опрос цен open-позиций через Birdeye, пороги TP/SL (константы), маршрут по
     `dexId`, при продаже — закрытие `Position` + запись `Trade` с `realizedPnlUSD` и
     `reason`.
- Блокчейн-модули (`raydium-buy`, `raydium-sell`, `pumpfunswap`) переезжают на бэк и
  **принимают `Keypair` параметром**, а не берут глобальный `owner` из env. Это позволяет
  подписывать кошельком конкретного пользователя.
- Источники: Birdeye — цены (платный API-ключ на сервере), DexScreener — кандидаты.

### 4. API (Express)

Новые/изменённые роуты (все, кроме auth, под `authMiddleware`):

- `POST /wallet` — сохранить/заменить зашифрованный кошелёк; возвращает только `publicKey`.
- `GET /wallet` — вернуть `publicKey` + `botEnabled` (никогда не секрет).
- `DELETE /wallet` — удалить кошелёк.
- `POST /bot/start`, `POST /bot/stop` — переключение `botEnabled`.
- `POST /sell/manual` — ручная продажа кошельком пользователя (обе биржи); тело: токен,
  количество, биржа.
- `GET /positions` — открытые позиции пользователя.
- `GET /trades` — закрытые сделки пользователя.

**Фиксы auth** (делаем, раз и так трогаем слой):
- `bcrypt.hash` rounds 3 → 12.
- Убрать `console.log(user)` в `login` (утечка хэша пароля в логи).
- Унифицировать опции cookie (`sameSite`, `secure`) между registration/login/refresh через
  единый конфиг.
- Убрать заглушку `/dashboard` (`getUsers` возвращает `['123','456','789']`) — заменить на
  реальные данные или удалить роут.

### 5. Фронтенд (минимально в этой фазе)

- `WalletGenerator` → форма «вставить приватный ключ»: секрет показывается один раз, уходит
  на бэк через `POST /wallet`, очищается из состояния. (Генерация нового кошелька опционально
  остаётся, но секрет тоже не задерживается в UI.)
- Кнопка Start/Stop бота → `POST /bot/start` | `/bot/stop`, состояние из `GET /wallet`.
- `ManualSellForm` → зовёт `POST /sell/manual` вместо локальных блокчейн-функций.
- **Удалить** клиентский движок `useAutoTrade` и торговое состояние в `localStorage`;
  `DashboardPage` читает `GET /positions` и `GET /trades`.
- Полностью убрать `VITE_PRIVATE_KEY` из фронтенда и связанный код (`raydium-buy-token.ts`,
  `pumpfunswap-buy.ts` и т.д. переезжают на бэк).

## Обработка ошибок

- Сбои RPC/swap логируются; позиция не портится. Ретраи с подъёмом `priorityFee` как сейчас.
- Неудачная продажа оставляет `Position` в статусе `open` (повтор на следующем проходе).
- Недостаток ликвидности (`INSUFFICIENT_LIQUIDITY`) → закрытие позиции с записью `Trade`
  и соответствующим `reason`.
- Ошибки шифрования/расшифровки — фатальны для конкретного кошелька: бот этого пользователя
  не торгует, ошибка логируется, остальные пользователи не затронуты.
- Воркер изолирован от API; авто-рестарт (dev: nodemon; прод: pm2 — ops-требование).

## Тестирование

- Юнит: round-trip шифрования (AES-256-GCM encrypt→decrypt), логика фильтра, расчёт PnL,
  математика цены/количества (нормализация decimals, lamports).
- Интеграционные: API-эндпоинты (CRUD кошелька, переключение бота, ручная продажа, чтение
  позиций/сделок) с `authMiddleware`.
- Движок: **dry-run режим** — симуляция покупок/продаж без реальных транзакций, на моках
  Birdeye/DexScreener, для проверки логики фильтра/входа/выхода.
- В ходе реализации — TDD.

## Допущения

1. Размер покупки, TP %, SL %, лимит позиций, кулдаун — на этой фазе **константы серверного
   конфига**; выносятся в UI в Под-проекте 4.
2. Шифрование серверное (мастер-ключ в env), без пароля-на-сессию — требование 24/7-торговли.
   Мастер-ключ живёт вне БД.
3. Один воркер-процесс, последовательный обход пользователей — достаточно для умеренного
   числа пользователей; путь роста — очередь (BullMQ+Redis) при необходимости.
4. Birdeye — источник цен, DexScreener — источник кандидатов (как сейчас).
5. Один кошелёк на пользователя в этой фазе.
