# Под-проект 4 — Редактируемый фильтр кандидатов (UI + БД, per-user)

**Дата:** 2026-06-18
**Статус:** утверждён к реализации
**Фаза:** 4 из 5 (вытянут вперёд по запросу пользователя)

## Контекст

В Под-проекте 1 серверный торговый движок фильтрует кандидатов по **глобальным
константам** в `backend/src/config/trading-config.ts` (`FILTER` + `TRADING.MAX_TOKEN_AGE_SEC`):
мин. ликвидность, макс. капитализация, мин. бусты, макс. возраст пары, разрешённые
биржи, чейн. Функция `passesFilter(candidate, nowMs)` читает эти константы напрямую.

Сравнение с оригинальным клиентским фильтром (`useAutoTrade.ts`, удалён) показало, что
расходится **только** `MIN_BOOSTS` (сервер `50` против оригинала `5000`); ликвидность
(25000), капитализация (1300000) и возраст (25 мин) идентичны.

Цель этого под-проекта: вынести фильтр кандидатов из констант в **редактируемые
per-user настройки**, хранимые в БД. Пользователь вводит значения в форме на странице
настроек, сохраняет, и видит отдельно отображённый «применённый фильтр». Движок читает
фильтр конкретного пользователя из БД вместо глобальных констант.

## Объём

Редактируемыми становятся **четыре** поля фильтра:
- мин. ликвидность (USD),
- макс. капитализация (USD),
- макс. возраст пары (в **минутах** в UI),
- мин. бусты.

**Не входит (остаётся константами этой фазы):**
- разрешённые биржи (`raydium`, `pumpswap`) и `chainId` (`solana`);
- торговые параметры (размер покупки, TP/SL, кулдаун, лимит позиций) — отдельные под-проекты.

## Решения (из брейншторма)

1. **Хранение — подход A:** отдельная коллекция `FilterConfig` (один документ на
   пользователя). Кошелёк и фильтр разделены (у пользователя отдельные планы по UI кошелька).
2. **Семантика значений — обязательные поля:** все 4 поля обязательны, конкретные числа.
   Концепции «0 = без ограничения» нет. Границы валидации защищают от значений,
   отсекающих всех кандидатов.
3. **Дефолт до настройки — бот не стартует без фильтра:** пока пользователь не сохранил
   фильтр, движок его не торгует, а кнопка Start заблокирована. Никакого авто-сидирования
   констант в движок.

## Компоненты

### 1. Модель данных — `FilterConfig` (Mongoose, новая коллекция)

```
user: ObjectId(ref User)   // уникальный индекс — один фильтр на пользователя
minLiquidityUSD: number
maxMarketCapUSD: number
maxAgeMinutes:   number
minBoosts:       number
updatedAt: Date
```

- Документ создаётся при первом сохранении (`PUT /filter`). Отсутствие документа =
  «фильтр не настроен».
- Хранятся «сырые» числа (без форматирования). Возраст хранится в **минутах** (как
  вводит пользователь); пересчёт в секунды — на стороне движка.

### 2. Валидация (сервис/контроллер, при `PUT /filter`)

Все 4 поля обязательны, числовые и конечные (`Number.isFinite`). Границы:

| Поле | Правило | Причина |
|------|---------|---------|
| `minLiquidityUSD` | `>= 0` | 0 = ликвидность не проверяется (всегда истина) |
| `maxMarketCapUSD` | `> 0` | `<= 0` отсёк бы всех кандидатов |
| `maxAgeMinutes` | `> 0`, целое | `0` отсёк бы всех кандидатов |
| `minBoosts` | `>= 0`, целое | 0 = любой уровень бустов |

Нарушение любого правила → `ApiError.BadRequest` (HTTP `400`) с понятным сообщением.

### 3. API (Express, все под `authMiddleware`)

- `GET /filter` — вернуть сохранённый фильтр пользователя, либо `null`, если не настроен.
  Никогда не отдаёт чужой фильтр (скоуп по `req.user.id`).
- `PUT /filter` — upsert фильтра пользователя (валидация → сохранение → возврат
  сохранённого документа).
- `POST /bot/start` — **новый гейт:** если у пользователя нет `FilterConfig`, вернуть
  `400` «Настройте фильтр перед запуском бота». Существующая проверка наличия кошелька
  сохраняется.

DELETE не нужен (YAGNI).

### 4. Движок (worker)

- Сигнатура меняется: `passesFilter(candidate, nowMs, filter)` — четыре редактируемых
  порога берутся из переданного `filter` вместо глобального `FILTER`. `chainId` и
  `ALLOWED_DEXES` по-прежнему из констант `trading-config`.
- `maxAgeMinutes` пересчитывается в секунды внутри сравнения (`maxAgeMinutes * 60`).
- В каждом buy-проходе (`runBuyPass`) движок загружает `FilterConfig` пользователя и
  передаёт в `passesFilter`. Нет записи → пользователь пропускается в этом проходе
  (страховка; в норме недостижимо, т.к. бот без фильтра не стартует).
- Изменение фильтра применяется на следующем проходе автоматически (движок читает БД
  каждый проход) — отдельного рестарта не требуется.

### 5. Фронтенд (`AccountSettingsPage`)

Новый компонент `FilterSettings`, размещается рядом с `BotControl`.

- **Форма редактирования** — 4 поля: мин. ликвидность ($), макс. капитализация ($),
  макс. возраст (минуты), мин. бусты.
- **Форматирование тысяч** запятыми в денежных полях при вводе (`10,000`,
  `1,300,000`); в состояние/на бэк уходит чистое число (парсинг — снятие нецифровых
  символов). Возраст и бусты — обычные целые без форматирования.
- Кнопка **Сохранить** → `PUT /filter`. Поля обязательны; клиентская валидация
  зеркалит серверные границы, но сервер — источник истины.
- **Панель «Применённый фильтр»** (read-only) рядом с формой — показывает текущие
  сохранённые значения (форматированные), визуально отделённая от полей редактирования.
- На загрузке `GET /filter`: если фильтр есть — заполняет и панель, и поля формы; если
  нет — поля пустые с плейсхолдерами, панель показывает «Фильтр не настроен».
- `BotControl`: кнопка Start заблокирована (с подсказкой), пока фильтр не сохранён.

## Обработка ошибок

- `PUT /filter` с невалидными данными → `400`, фильтр не меняется.
- `POST /bot/start` без фильтра → `400`, бот не включается.
- Движок: отсутствие/ошибка чтения `FilterConfig` для пользователя логируется, этот
  пользователь пропускается в проходе, остальные не затронуты.

## Тестирование (TDD)

- **Юнит (бэк):**
  - `passesFilter(candidate, nowMs, filter)` — граничные случаи по каждому из 4 полей
    (ровно на границе, чуть ниже/выше), плюс проверка пересчёта минут→секунд.
  - Логика валидации фильтра (каждое правило границ, отказ при пропущенном поле).
- **Интеграционные (бэк, HTTP):**
  - `GET /filter` / `PUT /filter` под `authMiddleware` (401 без токена).
  - **IDOR:** пользователь A не может прочитать или изменить фильтр пользователя B;
    скоуп строго по `req.user.id`. (Закрывает часть «хвоста» IDOR-тестов из Под-проекта 1.)
  - Гейт `POST /bot/start` без фильтра → `400`; с фильтром → `200`.
- **Фронт (минимально):** юнит на хелпер форматирования/парсинга тысяч.

## Допущения

1. Один фильтр на пользователя (как один кошелёк в текущей фазе).
2. Биржи и чейн не редактируются в этой фазе.
3. Возраст вводится и хранится в минутах; движок переводит в секунды.
4. Денежные поля форматируются запятыми только в UI; БД хранит числа.
